## 第一部分：系统整体架构（约 1.5 分钟）

*(配合 PPT 第一页：System Core Architecture)*

我们的系统采用经典的**三层架构**：

### 首先是 前端层（Frontend Layer）
我们构建了包含**仪表盘**、**持仓管理**、**交易记录**、**搜索功能**以及**AI 助手**在内的完整用户界面。
前端通过标准的 **HTTP REST API** 与后端进行交互，实现了前后端分离。

### 然后是 应用层（Application Layer）
我们采用了**Spring Boot 3.4.5** 框架来构架应用层。应用被分成三个模块：
- **Controller** 负责路由分发和请求处理
- **Service** 承载核心业务逻辑和数据验证
- **Repository** 处理数据访问操作

我们使用 **JPA / Hibernate** 作为 ORM 映射工具，极大地简化了对象与关系数据库之间的转换。

### 在 数据层（Data Layer）
我们选用 **MySQL 8.0** 作为持久化存储，并引入了 **Caffeine** 进行本地缓存，减少新浪API和数据库压力。

这种架构保证了系统模块化且易于扩展。

## 第二部分：业务逻辑分层详解

接下来深入系统的核心——业务逻辑分层。通过遵循职责分离原则，我们将复杂的股票交易流程拆解为清晰的逻辑链条。

**Controller 层**作为"调度员"，负责三大核心入口：
**HoldingController**，
**TransactionController**，
和 **StockController**。

有必要强调 Controller 层不包含具体业务逻辑，它只负责接收请求、参数校验、调用 Service 层并返回响应。所有的业务规则都在 Service 层实现。

服务层的核心也是三个部分
- **HoldingService** 是 ”资产管家“，负责投资组合的核心计算，包括成本核算、盈亏追踪、余额检查以及手续费计算（总费率 0.02%）
- **TransactionService**是一个”忠实记录员“，确保每一笔交易都有据可查，负责创建交易记录和查询统计
- **StockService**，一个市场联络员，负责对接外部市场数据，集成新浪 API，采用 UA 轮换和 3 次重试策略，并通过 Caffeine 缓存加速高频数据访问

整个业务处理流程形成严密的七步闭环：参数校验 → 获取实时报价 → 金额计算 → 余额检查 → 更新持仓 → 创建交易记录。

（手势强调，语速放慢）

一个最关键的设计点！我们在 HoldingService 类级别使用了 `@Transactional` 注解，并指定 `rollbackFor = Exception.class`。
这个注解确保了**事务的原子性和异常回滚**——任何异常都会导致整个事务回滚，彻底杜绝"部分成功"问题，比如钱扣了但股没到。

（可以这样强调，加重语气）

正是这一行代码，确保了我们的系统在面临网络波动或逻辑错误时，依然能像银行系统一样，严守"账实相符"的底线。

## 第三部分：然后是 完整数据流分析

当用户在界面上点击"买入"按钮时，会发生什么呢。 让我们快速梳理一下。

**第一步：用户行为** —— 用户在前端触发交互

**第二步：HTTP 请求** —— 浏览器发送 POST 请求到后端，包含股票代码、数量和日期

**第三步：控制器层** —— RestController 接收请求并分发给对应的 Service 方法

**第四步：服务层** —— Service 执行核心业务：获取实时股价、计算交易总额和手续费、检查现金余额

**第五步：持久层** —— JpaRepository 转化为 SQL 操作

**第六步：数据库执行** —— MySQL 执行 INSERT 或 UPDATE 语句

**第七步：响应封装** —— 后端将结果封装为 JSON 格式

**第八步：页面渲染** —— 前端解析 JSON，动态更新持仓列表和账户余额

这个数据流的特点：**单向流动**、**异常可控**、**性能优化**。

## 第四部分：数据库表结构设计

最后，我们来看支撑上述逻辑的基石——数据库表结构。

我们有两张核心表：

**Holding 表（持仓信息表）**：记录用户当前持仓状态，包含股票代码、数量、成本价等关键字段。

**Transaction 表（交易流水表）**：记录每一笔交易的历史记录，通过外键关联持仓表，记录交易类型、数量、价格等信息。

（手势比划，强调重点）

这是保证数据完整性的关键设计！Transaction 表通过外键 holdingId 关联到 Holding 表的主键 id，形成一对多的强关联关系。

这种设计有3大优势：**可追溯性**——每笔交易都能精确追溯到具体持仓；**数据完整性**——无法创建"孤儿"交易记录；**查询效率**——通过 holdingId 快速还原某个持仓的所有历史操作。

举个例子，当用户想查看某只股票的所有买卖记录时，只需三步：找到 Holding 表的对应 id → 在 Transaction 表查询该 id 的所有记录 → 即可得到完整的交易流水。
